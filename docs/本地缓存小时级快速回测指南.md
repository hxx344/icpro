# 本地缓存小时级快速回测指南

本文档用于说明如何基于本地小时级数据运行快速回测，并生成可复用的结果报表。

需要先澄清一点：实际回测入口虽然是从 `data/options_hourly` 读取原始小时级 Parquet，但真正加速回测的，不只是这层原始数据，而是程序内部还会自动建立一层更快的二级缓存。

也就是说，平时说“用小时级缓存跑得更快”，通常指的是：

1. 原始小时级数据：`data/options_hourly/<UNDERLYING>/*.parquet`
2. 自动生成的月度标准化缓存：`data/.cache/options_hourly_monthly/<UNDERLYING>/`
3. 自动生成的区间级磁盘缓存：`data/.cache/options_hourly_store/<UNDERLYING>/`

第一次跑会先构建这些缓存；后续同区间重复跑，速度会明显更快。

## 适用脚本

主入口脚本：`scripts/run_cached_hourly_backtest.py`

这个脚本的固定流程是：

1. 读取配置文件；
2. 自动检测 `data/options_hourly/<UNDERLYING>/*.parquet` 的真实覆盖区间；
3. 默认直接使用这段真实覆盖区间作为回测时间范围；
4. 直接基于本地小时级缓存运行回测；
5. 导出成交明细、汇总 JSON、标准图表，以及 PnL 直方图。

## 一、运行前准备

### 1. 确认本地已有小时级缓存

脚本会先从以下目录读取原始期权小时数据：

- `data/options_hourly/BTC/*.parquet`
- 或其他标的对应目录

如果目录下没有 parquet 文件，脚本会直接报错。

另外，程序在运行过程中会自动维护两层加速缓存：

- `data/.cache/options_hourly_monthly/<UNDERLYING>/`
- `data/.cache/options_hourly_store/<UNDERLYING>/`

这两层缓存不需要手工指定，回测时会自动命中。

### 2. 确认 Python 环境可用

在项目根目录下执行，并确保依赖已安装。

Windows 常见方式：

```powershell
.\.venv-1\Scripts\Activate.ps1
```

## 二、最常用的运行方式

### 1. 默认方式

如果不传参数，脚本会默认使用：

- 配置文件：`configs/backtest/weekend_vol_btc_hourly.yaml`
- 时间范围：本地小时缓存的真实覆盖区间

命令：

```powershell
python scripts/run_cached_hourly_backtest.py
```

### 2. 显式指定 backtest 配置

```powershell
python scripts/run_cached_hourly_backtest.py --config configs/backtest/weekend_vol_btc_hourly.yaml
```

适合场景：

- 你已经有标准回测 YAML；
- 想严格按某份 backtest 配置复现结果。

### 3. 直接使用 trader 配置

```powershell
python scripts/run_cached_hourly_backtest.py --config configs/trader/weekend_vol_btc.yaml
```

适合场景：

- 你想直接验证当前交易程序参数；
- 希望尽量减少“实盘参数”和“回测参数”之间的映射误差。

脚本会自动把 trader 配置映射成 backtest 配置后再运行。

### 4. 手动限制回测区间

```powershell
python scripts/run_cached_hourly_backtest.py --start 2024-01-01 --end 2024-12-31
```

说明：

- `--start` 和 `--end` 都是可选的；
- 格式为 `YYYY-MM-DD`；
- 如果你填的区间超出本地缓存覆盖范围，脚本会自动裁剪到真实可用区间。

### 5. 指定输出目录

```powershell
python scripts/run_cached_hourly_backtest.py --output-dir reports/my_hourly_backtest
```

适合场景：

- 你不想覆盖默认输出目录；
- 你要同时对比多组参数结果。

## 三、脚本参数说明

脚本支持以下参数：

- `--config`
  - 配置文件路径
  - 默认值：`configs/backtest/weekend_vol_btc_hourly.yaml`
- `--start`
  - 手动起始日期，格式 `YYYY-MM-DD`
  - 默认：使用缓存真实起点
- `--end`
  - 手动结束日期，格式 `YYYY-MM-DD`

- `--output-dir`
  - 自定义报表输出目录
  - 默认：使用配置文件中的 `report.output_dir`

## 四、脚本内部的几个关键行为

### 0. 真正加速的是内部二级缓存

这部分是前一版文档漏掉的关键点。

`scripts/run_cached_hourly_backtest.py` 本身会把 `option_data_source` 设为 `options_hourly`，但底层不是每次都重新把原始月度 parquet 全量慢读一遍。

底层加载流程是：

1. 先按区间读取 `data/options_hourly/<UNDERLYING>/*.parquet`
2. 自动把月文件标准化后缓存到 `data/.cache/options_hourly_monthly/<UNDERLYING>/`
3. 再把当前回测区间对应的数据集缓存到 `data/.cache/options_hourly_store/<UNDERLYING>/`
4. 下次同样区间再跑时，优先直接命中磁盘缓存

因此：

- 第一次跑：会慢一些，因为要建缓存；
- 第二次开始：通常会快很多；
- 同一个标的、同一个时间区间重复测试参数时，这套缓存最有价值。

相关实现位置：

- [src/options_backtest/data/hourly_store.py](src/options_backtest/data/hourly_store.py#L145-L162)
- [src/options_backtest/data/hourly_store.py](src/options_backtest/data/hourly_store.py#L295-L425)
- [src/options_backtest/data/loader.py](src/options_backtest/data/loader.py#L259-L275)
- [scripts/run_cached_hourly_backtest.py](scripts/run_cached_hourly_backtest.py#L786-L812)

### 1. 自动识别真实数据覆盖范围

脚本会扫描原始小时级目录：

- `data/options_hourly/<标的>/*.parquet`

然后读取首尾文件中的 `timestamp`，得到真实覆盖区间。

这意味着：

- 不需要手工先查数据起止时间；
- 默认回测范围总是和本地缓存一致；
- 很适合快速复盘和批量试参数。

但这里的“本地缓存”更准确地说，是“先根据原始 `options_hourly` 识别覆盖范围，再由内部二级缓存负责加速加载”。

### 2. 自动兼容两类配置

脚本支持两种配置来源：

- 标准 backtest YAML
- trader YAML

如果传入的是 trader 配置，脚本会自动映射出回测所需字段，例如：

- `target_delta`
- `wing_delta`
- `default_iv`
- `entry_day`
- `entry_time_utc`
- `leverage`
- `quantity`
- `compound`
- `max_delta_diff`
- `stop_loss_pct`
- `stop_loss_underlying_move_pct`

### 3. trader 配置会自动按 USD 初始资金换算

当使用 trader 配置时，脚本内部默认按：

- 初始资金 `10,000 USD`

并在回测开始时，按照首个现货价格折算成币本位初始余额。

这能让“当前交易程序配置”回测时更接近实盘资金口径。

### 4. 会同时输出两套结果

一次运行会输出两组数据集：

#### `observed_hour`
- 含义：按小时观测到的止损触发结果直接成交
- 用途：更贴近“小时级缓存能看到的实际路径”

#### `cap_at_threshold`
- 含义：把超出止损阈值的损失，截断到止损阈值本身
- 用途：用于对比“观察到的小时止损穿透”与“理想止损阈值成交”之间的差异

## 五、输出结果会生成什么

输出目录下通常会包含：

### 1. 成交明细 CSV

例如：

- `trades_起始日_结束日.csv`

内容通常包括：

- 开仓时间
- 平仓时间
- 合约名
- 数量
- PnL
- 开仓前余额
- 开仓前权益
- PnL 占当前余额百分比
- 止损原因等附加诊断字段

### 2. 汇总 JSON

例如：

- `backtest_summary_起始日_结束日.json`

里面会包含：

- 配置来源
- 数据覆盖区间
- 实际运行区间
- 交易统计
- 胜率
- 总 PnL
- 各类绩效指标
- 图表路径

### 3. 图表 HTML

通常包括：

- 标准权益曲线等图表
- `pnl_histogram_*.html`
- `pnl_histogram_profit_loss_*.html`
- `pnl_histogram_by_year_*.html`

### 4. 止损对比文件

如果存在止损封顶调整，还会生成：

- `stop_loss_cap_details_起始日_结束日.csv`
- `backtest_comparison_起始日_结束日.json`

用于对比：

- `observed_hour`
- `cap_at_threshold`

两种口径下的回测差异。

## 七、推荐使用方式

### 场景 0：先热启动一次缓存

如果很久没跑过，或者刚更新过 `data/options_hourly` 原始文件，建议先跑一次完整回测，让内部缓存先生成。

例如：

```powershell
python scripts/run_cached_hourly_backtest.py --config configs/trader/weekend_vol_btc.yaml
```

之后再重复跑相同区间，通常会比第一次更快。

### 场景 1：快速验证当前交易程序参数

直接运行：

```powershell
python scripts/run_cached_hourly_backtest.py --config configs/trader/weekend_vol_btc.yaml
```

适合：

- 看当前实盘参数过去一段时间表现；
- 做实盘参数变更前复核。

### 场景 2：只想吃满本地缓存区间

直接运行：

```powershell
python scripts/run_cached_hourly_backtest.py
```

适合：

- 快速跑全样本；
- 不想手工填日期。

### 场景 3：做某一年或某一段窗口复盘

```powershell
python scripts/run_cached_hourly_backtest.py --config configs/trader/weekend_vol_btc.yaml --start 2025-01-01 --end 2025-12-31
```

适合：

- 单年表现拆分；
- 某个波动阶段专项复盘；
- 对比不同年份策略稳定性。

### 场景 4：批量试参数时区分输出目录

```powershell
python scripts/run_cached_hourly_backtest.py --config configs/trader/weekend_vol_btc.yaml --output-dir reports/validation/weekend_vol_run_a
```

适合：

- 多次试验不互相覆盖；
- 后续做汇总比较。

## 八、常见问题

### 1. 为什么我明明传了日期，结果还是被改了？

因为脚本会把输入日期裁剪到本地缓存的真实覆盖区间内。

也就是说：

- 早于缓存起点的日期会被截到缓存起点；
- 晚于缓存终点的日期会被截到缓存终点。

### 2. 为什么默认回测范围不是配置 YAML 里的起止时间？

因为这个脚本的设计目标就是：

- 优先使用本地小时缓存的真实覆盖区间；
- 避免忘记数据真实边界；
- 减少“配置写了但本地其实没数据”的问题。

### 3. 为什么会有 `observed_hour` 和 `cap_at_threshold` 两套结果？

因为小时级缓存回测里，止损在小时级别可能出现“穿透”。

所以脚本会同时给出：

- 观察到的实际小时止损结果
- 把亏损封顶到止损阈值后的结果

便于横向比较。

### 4. 没有 parquet 文件怎么办？

需要先准备本地小时级缓存，否则脚本无法运行。

### 5. 为什么我记得之前是“用一个单独缓存跑得更快”？

这个记忆是对的。

更准确地说，回测有两层数据：

- 原始小时级数据：`data/options_hourly/...`
- 自动生成的加速缓存：`data/.cache/options_hourly_monthly/...` 和 `data/.cache/options_hourly_store/...`

所以之前说的“单独的小时级缓存”大概率就是在指这两层 `data/.cache/` 下的加速缓存，而不是最原始的 `options_hourly` 月度 parquet 本身。

如果这些缓存已经建好，再重复跑相同标的/区间，速度会明显更快。
